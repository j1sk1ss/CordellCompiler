\begin{abstract}
\par This report summarizes the documented design of the Cordell Programming Language (\cpl) and the Cordell Compiler, its restricted grammar and advanatages that are reached. The report covers the language \textbf{core}, preprocessing, tokenization and markup, \textbf{AST} and \textbf{IR} generation, CFG and SSA construction, DAG-based optimization, liveness analysis, register allocation, copy propagation, IR lowering, peephole optimization, code generation, static diagnostics, debugging, and the documented language benchmark values against GCC 14.2.0 and Apple Clang 12.0.0.
\end{abstract}

\section{Introduction}
\par \cpl{} is documented as a small systems language and compiler project intended mainly for learning compiler architecture, experimenting with optimizations, and producing low-level code. The project is also connected with the development of operating systems, IoT-applications, etc.
\par The project is not presented as a feature-rich general-purpose language. The documentation explicitly emphasizes low-level behavior, C-like syntax, assembly affinity, and a restricted language core. The language omits structures, classes, enums, and unions. This restriction is not accidental: it keeps the parser and language surface small so that the compiler pipeline, optimizations, and backend experiments remain the central part of the project.

\section{Design Goals}
The documentation frames \cpl{} around five practical goals.

\begin{itemize}
  \item \textbf{Simplicity:} the language should stay small and explicit.
  \item \textbf{No safety:} CPL does not attempt to provide memory safety; it prefers low-level control and diagnostics.
  \item \textbf{Systems scope:} intended programs include bootloaders, edge DBMSs, edge file systems, primitive compilers, and small interpreters.
  \item \textbf{Experiments:} the compiler is a test stand for optimizations, symbol-table experiments, perceptron-based heuristics, SMT ideas, and peephole DSLs.
  \item \textbf{Education:} the repository documents a complete compiler-building journey from preprocessing to code generation.
\end{itemize}

\section{Language Core}
\subsection{Types}
\cpl{} is a permissively statically typed language. Values do not freely change type, but the compiler may insert safe widening conversions. Narrowing conversions are never implicit and must be written with \code{as}. The same explicit cast syntax is also useful when selecting an overloaded function.

Primitive types include signed and unsigned integers from 8 to 64 bits, \code{f32}, \code{f64}, and \code{i0}. The \code{i0} type is a void-like type for function returns and \code{ptr i0}. The language does not have a separate Boolean type: zero means false and non-zero means true.

\begin{lstlisting}[caption={Explicit casts and pointer operations.}]
i32 a = 10 as i32;
u8  b = (a + 20 as i32) as u8;
ptr i32 p = ref a;
dref p = 11 as i32;
\end{lstlisting}

The main non-primitive containers are \code{str} and \code{arr}. A stack string behaves similarly to an array of \code{i8} with a terminating zero byte. Independent string literals are stored in a read-only section and must be referred to with \code{ref}. Arrays contain primitive element types and are used with explicit indexing and pointer-style operations.

\subsection{Control flow and operations}
The documented operation set includes arithmetic, comparisons, update operations, bit operations, and logical operations. Logical \code{\&\&} and \code{||} support lazy evaluation, while \code{@[not\_lazy]} forces both sides to be generated.

The control-flow syntax is C-like but intentionally explicit. An \code{if} condition is followed by a semicolon. \code{while} is the ordinary conditional loop. \code{loop} is an endless loop; it must contain \code{break} or use \code{@[counter(...)]}. \code{switch} supports C-style fallthrough, while \code{@[no\_fall]} adds hidden breaks to cases.

\begin{lstlisting}[caption={Counted loop and non-fallthrough switch.}]
@[counter(10)] loop {
    : repeated ten times :
}

@[no_fall]
@[straight]
switch tag; {
    case 1; { exit 1; }
    default { exit 0; }
}
\end{lstlisting}

\subsection{Functions}
Functions are declared with \code{function}. Prototypes, default arguments, overloading, local functions, function pointers, lambda functions, and variadic arguments are documented. Overloading is restricted: return-type-only overloading is forbidden, overloads are local-only, and candidates are searched in the same scope. Function pointers use \code{ptr i0}, but no signature information remains available through such a pointer.

Local functions and lambdas do not capture the parent environment. Therefore they are lightweight logic containers, not full closures.

\FloatBarrier
\begin{lstlisting}[caption={Local function passed as a pointer.}]
function apply(ptr i0 f, i32 a, i32 b) {
    f(a, b);
}

@[entry("_start")] function main() {
    function add(i32 a, i32 b) {
        return a + b;
    }
    apply(add, 10 as i32, 20 as i32);
    exit 0;
}
\end{lstlisting}

Two built-in mechanisms expose low-level behavior: \code{syscall} and \code{asm}. A syscall is invoked like a function and takes a platform-dependent argument list. \section{Front End}
The preprocessor handles \code{include}, \code{define}, \code{undef}, \code{ifdef}, and \code{ifndef}. This is necessary because the project uses C-like header organization: headers contain prototypes and guards, while implementation files contain bodies. After preprocessing, \code{line} directives preserve source-origin information for tokenization.

Tokenization splits the input byte sequence into basic tokens using DFA-like logic. Markup is a second token-level stage that uses scopes and declarations to attach semantic information to tokens. If \code{i32 a} is declared in a scope, later visible occurrences of \code{a} can be marked as \code{i32}.

The parser then builds an AST using an LL approach. AST-level optimizations include condition unrolling and dead-scope elimination, but the documentation treats later IR-level optimizations as the main optimization path.

\section{HIR, CFG, and SSA}
The AST is flattened into HIR, a three-address-code-like representation. This makes later optimization easier. For example, array indexing is expanded into base-address computation, scaled offsets, address addition, and memory store operations.

The CFG is generated from HIR. The documentation describes both one-block-per-command and leader-based splitting. The leader approach starts a block at the first instruction, at jump targets, and immediately after jumps.

SSA form then renames assigned variables so that each assignment creates a unique variable version. Dominators, strict dominators, and dominance frontiers are computed to place phi functions where control-flow paths merge.

\section{Optimization}
After SSA, the compiler can build a DAG that records how values are derived. Nodes with the same hash and compatible dominance can be merged, enabling common subexpression elimination. HIR optimizations also include constant folding and propagation, dead function elimination via a call graph, tail recursion elimination, function inlining, loop canonicalization, and LICM.

\section{LIR, Liveness, and Registers}
LIR is a lower intermediate representation closer to target instructions. It is similar to 3AC but uses a two-address form. It bridges HIR and the target-specific assembly dialect. On x86-64, instruction selection, ABI-related pre-coloring, and register allocation become important at this level.

Liveness analysis computes \code{USE}, \code{DEF}, \code{IN}, and \code{OUT} for each CFG block:
\[
OUT[B] = \bigcup_{S \in succ(B)} IN[S]
\]
\[
IN[B] = USE[B] \cup (OUT[B] - DEF[B]).
\]
The compiler iterates in reverse order until the sets stabilize. If a variable appears in \code{USE} or \code{DEF} but not in \code{OUT}, the compiler can mark it dead with a special \code{kill} instruction.

For pointers, the compiler uses an alias map. If \code{b0} owns a reference to \code{a0}, \code{a0} must not be killed while \code{b0} is live. This is stack-slot reuse and liveness protection, not Rust-style memory safety.

Register allocation builds an interference graph. Vertices are variables; edges connect variables that are live at the same time. Coloring assigns different colors to adjacent nodes, and each color corresponds to a reusable physical register choice.

\section{Peephole Optimization and Codegen}
The peephole optimizer performs pattern matching and replacement on LIR or assembly-like forms. The simplest documented example is replacing \code{mov rax, 0} with \code{xor rax, rax}. The project uses a PTRN DSL to generate pattern matchers instead of coding every pattern by hand.

The documented peephole stage has three passes: pattern matching, value propagation through redundant move chains, and dead-write cleanup. Code generation then unwraps special LIR operations such as \code{EXITOP} and \code{STRT} and emits NASM assembly.

\section{Static Diagnostics and Debugging}
The static checker has AST and IR parts. AST warnings cover read-only updates, declarations without initialization, wrong declaration values, missing returns, missing \code{exit} in \code{start}, wrong argument counts, wrong argument types, unused function results, return-type mismatches, illegal array access, duplicated branches, reserved function names, dead code, suspicious conversions, inefficient \code{while 1}, invalid \code{exit}, targetless \code{break}, misuse of \code{i0}, unused expressions, invalid \code{ref}, and non-even alignment.

IR analysis handles lower-level warnings such as null dereference, null dereference through arguments, constant conditions, propagated constants, wrong casts, and syscall-related checks. Static analysis is off by default and enabled with \code{--static-analysis}.

For debugging, the compiler can produce assembly with \code{--emit-asm}. The \code{-d} flag includes DWARF information, and \code{lis} is an interrupt point usable with \code{gdb} or \code{lldb}. The documentation recommends disabling optimizations when debugging code that must not be transformed.

\par The documentation warns that some C microbenchmarks use \code{asm volatile} to prevent dead-code elimination, while CPL preserves such loops by default. These tests should therefore be interpreted as loop-overhead measurements rather than broad compiler-performance benchmarks. The documentation also suggests reading CPL times conservatively, with about +30\% adjustment for possible measurement issues.

\section{Limitations}
This report is not an independent validation. It does not run the compiler, inspect source code, or reproduce benchmarks. All feature descriptions and measurements come from the uploaded documentation. The diagrams are TikZ reconstructions based on the documentation and reference images.

\section{Conclusion}
The documentation presents CPL as a compact, explicit, low-level language and Cordell Compiler as an end-to-end compiler laboratory. The pipeline covers preprocessing, tokenization, semantic markup, AST construction, HIR lowering, CFG and SSA, DAG-based optimization, LIR, liveness, register allocation, peephole optimization, and NASM code generation. The benchmark values suggest that the documented optimizer can generate compact and competitive code for selected small examples, while the most important contribution of the project is the coherent educational compiler pipeline.
